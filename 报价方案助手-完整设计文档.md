- 产品名称：报价方案助手

- 适用阶段：从 0 到 MVP 首发

## 2. 产品概述

### 2.1 产品定位

报价方案助手是一款面向网站/小程序外包团队的轻量化 SaaS 工具，用于将零散的客户需求快速整理为标准化、专业化的报价单与方案书，并支持在线预览、分享和 PDF 导出。

### 2.2 一句话介绍

10 分钟生成专业报价单和方案书，统一模板、自动合计、支持导出 PDF 和客户分享。

### 2.3 核心价值

1. 提升报价效率，减少重复文档劳动。
2. 统一团队对外输出形象。
3. 降低漏项、错项、价格计算错误的概率。
4. 让中小外包团队用更低成本拥有标准化售前工具。

### 2.4 目标用户

首版只聚焦以下用户，不做泛化：

1. 网站开发外包团队
2. 小程序开发团队
3. 独立开发者 / 接私单的前端或全栈
4. 小型数字化服务工作室

### 2.5 不做什么

为了确保 MVP 快速落地，首版不覆盖以下方向：

1. 不做多行业通用模板市场
2. 不做 CRM 全功能替代
3. 不做在线合同签署
4. 不做复杂的团队协同和审批流
5. 不做支付闭环
6. 不做深度 ERP/飞书/企业微信集成

## 3. 背景与问题定义

### 3.1 用户现状

目标用户当前通常这样工作：

1. 客户通过微信、电话、文档、语音发来零散需求。
2. 销售或项目经理将需求手工整理到 Word、Excel、Notion 或旧模板中。
3. 从历史项目复制旧报价，修改客户名、模块、价格和工期。
4. 使用不统一模板导出给客户。
5. 客户要求调整后继续手工修改。

### 3.2 现存问题

1. 报价效率低，制作一份方案通常要 30 分钟到数小时。
2. 多人使用不同模板，文档风格不统一。
3. 报价项漏填、价格算错、模块描述混乱。
4. 同类项目重复劳动严重。
5. 导出体验差，给客户的文档不够专业。
6. 客户要二次修改时，版本管理麻烦。

### 3.3 机会判断

这是一个高频、刚需、结果导向型场景，价值非常直接：

1. 节省时间
2. 降低出错率
3. 让文档看起来更专业
4. 提高成交前的信任感

这类需求不依赖大规模流量，适合个人开发者从窄场景切入。

## 4. 产品目标

### 4.1 商业目标

1. 在 4 到 8 周内做出可收费 MVP。
2. 2 个月内完成首批 10 到 30 个真实种子用户验证。
3. 3 个月内验证用户是否愿意为月订阅付费。

### 4.2 用户目标

1. 用户能在 10 分钟内生成一份可发送给客户的报价方案。
2. 用户无需 Word/Excel 即可完成从录入到导出。
3. 用户能复用历史项目，减少重复劳动。

### 4.3 MVP 成功标准

满足以下目标之一即可视为 MVP 成功：

1. 至少 5 个真实用户重复使用超过 3 次
2. 至少 3 个真实付费用户
3. 超过 50% 的注册用户完成“创建项目 -> 导出或分享”的完整闭环

## 5. 用户画像

### 5.1 画像 A：外包团队负责人

- 角色：老板 / 销售 / 项目经理
- 目标：快速给客户报价，提高成单效率
- 痛点：方案总临时赶，格式不统一，价格计算容易错
- 付费意愿：中高

### 5.2 画像 B：独立开发者

- 角色：个人接单者
- 目标：输出更专业，提高客户信任
- 痛点：不会做漂亮商务文档，报价没有标准模板
- 付费意愿：中

### 5.3 画像 C：工作室售前

- 角色：商务 / 售前顾问
- 目标：提升效率、减少重复编辑
- 痛点：每周写很多同质化方案
- 付费意愿：中高

## 7. 用户核心流程

### 7.1 主流程

1. 用户注册/登录
2. 进入项目列表
3. 点击新建项目
4. 填写客户基本信息
5. 填写项目说明
6. 新增/编辑报价项
7. 选择模板
8. 预览方案
9. 导出 PDF 或生成分享链接
10. 客户反馈后再次编辑

### 7.2 复用流程

1. 用户从项目列表中复制历史项目
2. 修改客户信息和报价项
3. 重新生成方案

### 7.3 AI 辅助流程

1. 用户录入项目类型、模块和客户行业
2. 点击“AI 生成项目简介”
3. 系统生成商务风格文案
4. 用户可继续编辑并保存

1. Web 预览页 + print CSS

#### 首版不做

1. 密码保护
2. 链接过期
3. 客户留言反馈
4. 查看日志分析

1. AI 生成项目简介
2. AI 生成服务范围说明

1. 项目类型
2. 客户行业
3. 功能模块

#### 输出要求

1. 语气专业
2. 商务风格
3. 不夸大承诺
4. 可人工编辑

#### 设计原则

AI 只是辅助，不接管核心业务逻辑。

## 12. 数据库设计

### 12.1 users

```sql
users (
  id uuid primary key,
  email varchar(255) not null unique,
  created_at timestamp not null,
  updated_at timestamp not null
)
```

### 12.2 projects

```sql
projects (
  id uuid primary key,
  user_id uuid not null references users(id),
  title varchar(255) not null,
  client_name varchar(255) not null,
  client_company varchar(255),
  project_type varchar(100) not null,
  contact_name varchar(100),
  contact_phone varchar(100),
  summary text,
  background text,
  goal text,
  scope text,
  duration varchar(100),
  delivery_note text,
  remark text,
  template_type varchar(50) not null default 'simple',
  total_price numeric(12,2) not null default 0,
  status varchar(50) not null default 'draft',
  share_token varchar(255),
  created_at timestamp not null,
  updated_at timestamp not null
)
```

### 12.3 quote_items

```sql
quote_items (
  id uuid primary key,
  project_id uuid not null references projects(id) on delete cascade,
  name varchar(255) not null,
  description text,
  unit_price numeric(12,2) not null default 0,
  quantity integer not null default 1,
  unit varchar(50),
  subtotal numeric(12,2) not null default 0,
  sort_order integer not null default 0,
  created_at timestamp not null,
  updated_at timestamp not null
)
```

## 18. 关键技术方案

### 18.1 实时预览方案

方案：

1. 编辑区使用表单状态管理
2. 预览区读取同一份状态
3. 所见即所得渲染模板组件

建议：

1. 使用 React Hook Form 或 Zustand 管理本地状态
2. 在保存前先在客户端完成计算和预览

### 18.2 金额计算方案

规则：

1. 小计 = 单价 x 数量
2. 总价 = 所有小计之和
3. 金额保留两位小数

### 18.3 PDF 导出方案

首版：

1. 文档预览页即打印页
2. 单独编写 print CSS
3. 隐藏不必要按钮和表单区

### 18.4 分享页方案

1. 创建分享 token
2. 通过 token 查询只读项目内容
3. 不暴露编辑接口

### 18.5 AI 生成方案

1. 前端发送结构化输入
2. 服务端拼装 Prompt
3. 限制输出长度和语气
4. 返回文本供用户确认后使用

## 24. 风险评估

### 24.1 产品风险

1. 场景过窄导致用户量有限
2. 场景过宽导致产品失焦
3. 用户认为“用 Word 也行”

### 24.2 技术风险

1. PDF 打印样式不稳定
2. AI 输出质量不稳定
3. 模板渲染复杂度逐渐上升

### 24.3 商业风险

1. 用户愿意用但不愿意付费
2. 缺少稳定获客渠道
3. 功能容易被模仿

### 24.4 应对策略

1. 先做窄场景，先打网站/小程序外包团队
2. 强化模板专业度和流程效率
3. 先验证付费，再扩功能
4. 提供历史项目复制和标准报价项库，提高迁移成本
